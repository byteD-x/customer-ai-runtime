# 业务需求分析

## 1. 项目定义

本项目建设一个“面向真实业务场景、支持行业适配增强、支持宿主系统挂载集成、支持插件化扩展的智能客服平台”。平台定位是企业客服能力层，而不是单一问答应用。

平台必须同时支持：

- 文本客服
- 语音客服
- RTC 实时语音通话客服
- RAG 知识增强
- 实时业务数据增强
- 行业适配增强
- AI 与人工协同
- 成本与缓存治理
- RAG 质量评测
- 人工接管队列
- 多渠道接入
- 多厂商适配
- 宿主系统挂载集成
- 宿主登录态桥接
- 插件化扩展

## 2. 当前事实与目标状态

### 2.1 当前事实

- 当前仓库已具备文本/语音/RTC/知识库/业务工具/人工接手/API/示例/测试能力。
- 当前仓库已落地 `Auth Bridge`、插件平台、行业增强联合编排、多认证桥接、宿主上下文解析、成本治理、本地模型价格表、RAG eval 本地标注样例、结构化交接包与单实例人工接管队列。
- 当前仓库默认使用本地 provider 可完整演示；真实 OpenAI / Qdrant / 外部业务系统 / 客服工单系统联调依赖外部配置，readiness 脚本只检查配置与健康状态。

### 2.2 Target State

后续目标状态包括：

- 多实例人工接管队列，例如 Redis sorted set 或数据库事务认领
- 真实 provider usage、租户预算、币种和账单周期驱动的成本结算
- 真实业务标注集和灰度流量驱动的 RAG 线上评估

## 3. 要解决的真实业务问题

1. 常见咨询重复高，人工客服成本高。
2. 订单、物流、售后、账号、工单等动态查询依赖多个业务系统，回复慢且易错。
3. 官网、App、小程序、H5、语音和通话入口割裂，体验不一致。
4. 单纯 FAQ 或 RAG 不能覆盖实时业务数据和高风险流程。
5. 用户要求人工、低置信度、投诉或敏感问题需要平滑转人工。
6. 宿主系统往往已有登录态和权限体系，不能强制重建认证体系。
7. 平台需要长期可维护、可灰度、可排障、可监控、可演进。
8. 复杂咨询存在主题切换与回退，系统不能把多轮会话当成一次性单问单答。
9. 高频 FAQ 需要低成本处理，但实时业务数据不能被错误缓存。
10. RAG 需要离线质量评测闭环，不能只靠成功 demo 证明效果。
11. 转人工需要运营侧队列管理，而不是只有“已转人工”的提示。

## 4. 必须覆盖的业务场景

### 4.1 文本客服

- 官网在线客服
- App 内客服
- 小程序 / H5 客服
- FAQ / 帮助中心问答
- 工单提交前智能问答
- 同一句短问在不同页面场景下触发不同路由

### 4.2 语音客服

- App 内语音咨询
- 语音输入与语音回复
- 非通话型语音交互

### 4.3 RTC 实时通话客服

- RTC 房间创建、加入、退出
- 用户音频流输入
- ASR 实时转文本
- RAG / 工具 / LLM 生成回复
- TTS 实时语音输出
- 打断、超时、重试、转人工、结束通话

### 4.4 人工协同

- AI 无法处理转人工
- 用户主动要求人工
- 低置信度转人工
- 高风险 / 投诉类转人工
- 向人工交接摘要、意图、历史和推荐回复
- 按技能组、优先级和入队时间查看等待队列
- 管理端认领下一条待处理会话

### 4.5 业务查询 / 工具调用

- 订单状态
- 售后进度
- 物流信息
- 账号信息
- 工单信息
- 课程 / 权限 / 套餐 / 运单 / CRM 信息

### 4.6 行业增强

- ecommerce
- saas
- education
- logistics
- crm / service
- custom

### 4.7 宿主系统挂载

- iframe / SDK / Web Component / API 中间层
- 复用宿主登录态与权限
- 传入用户、租户、页面、业务对象上下文

### 4.8 运营管理

- 知识库导入与更新
- Prompt / Policy 管理
- 会话检索
- 指标分析
- 成本摘要：provider、route、token、usage 来源、币种、账期、本地预算阈值、基于本地模型价格表的估算成本、缓存命中
- RAG eval：dataset、cohort、人工复核状态、route、引用关键词、有效命中、`offline_accuracy`、失败明细
- 故障排查
- 灰度与回滚

## 5. 业务增强原则

必须严格区分以下四类信息：

1. 通用静态知识：FAQ、规则、帮助文档。
2. 行业静态知识：订单规则、SaaS 权限规则、课程规则、物流规则等。
3. 实时业务数据：订单状态、库存、会员等级、物流轨迹、工单状态、学习进度等。
4. 当前会话与页面上下文：页面、订单号、商品 ID、用户身份、最近行为、会话摘要。

处理原则：

- 静态知识主要通过 RAG 处理。
- 实时业务数据必须优先通过业务 API / 工具插件获取。
- 当前上下文由宿主系统注入或上下文构造器动态生成。
- 路由决策必须同时参考消息文本、`page_context`、`business_objects` 和会话 `intent_stack`。
- 当路由置信度不足时，不直接强答，优先走澄清兜底或转人工。
- 知识问答可在版本、引用和 prompt 受控时缓存；实时业务查询不缓存。
- RAG 质量评测必须区分“有引用”和“有效命中”，低分兜底引用不能算准确命中。
- 最终回复必须融合行业知识、实时业务数据和当前会话上下文。

## 6. 核心角色

### 6.1 最终用户

- 发起文本、语音、RTC 会话
- 查看 AI 回复或请求人工

### 6.2 人工客服

- 接收转人工会话
- 查看摘要、意图、历史、推荐回复
- 继续处理复杂问题

### 6.3 运营人员

- 管理知识库、Prompt、策略、插件配置
- 观察效果指标与故障诊断

### 6.4 集成人员 / 平台研发

- 接入宿主系统
- 注册 Auth Bridge
- 注册行业适配器和业务工具插件
- 维护模型、ASR、TTS、RTC、向量库、业务 API 适配

## 7. 成功指标

指标必须标明口径：

- AI 首轮解决率：首轮无需人工结束会话的比例
- 总体解决率：AI 与人工协同后的闭环解决比例
- 转人工率：全部会话进入人工的比例
- 知识命中率：知识型问题检索到有效引用的比例
- 业务工具成功率：业务查询调用成功比例
- 文本响应时延：p50 / p95，首包与整句分别统计
- 语音首包时延：ASR 首结果或 TTS 首片段返回的 p50 / p95
- RTC 轮次成功率：单轮通话从音频输入到音频回推的成功比例
- 知识问答缓存命中率：知识类请求中命中安全缓存的比例
- 单轮估算成本：按 provider / route 聚合 token、usage 来源、币种、账期、本地预算阈值与估算成本
- RAG eval 通过率 / `offline_accuracy`：本地标注 case 中 route、引用关键词和有效命中均通过的比例
- 人工接管等待时长：按技能组统计 waiting_human 到 claim-next 的等待时间；当前 `atomic_claim` 只代表单进程锁内认领

## 8. 最小安全模型

1. 多租户隔离：任意请求必须落到 `tenant_id`，知识库、会话、房间、工具、插件配置都必须有租户边界。
2. 身份边界分离：平台访问认证、宿主用户身份认证、宿主业务权限校验、客服内部会话身份上下文必须明确分离。
3. 敏感信息保护：日志、错误、诊断、LLM 请求上下文必须脱敏。

## 9. 非功能要求

- 可集成：支持 API、SDK、挂载、子应用、进程内 facade。
- 可扩展：插件化扩展路由、工具、行业、鉴权、上下文和回复后处理。
- 可替换：LLM、ASR、TTS、RTC、向量库、业务 API 支持适配层替换。
- 可观测：日志、诊断、指标、关键状态变化可追踪。
- 可解释：路由结果需要输出 `confidence`、`confidence_band`、`intent` 与命中信号。
- 可验证：单元测试、集成测试、关键链路测试、挂载与鉴权桥接测试。
- 低成本：默认本地 provider 可跑，外部付费服务作为可选增强。
- 可复现：面试演示、RAG eval、外部 readiness 和完整测试门禁都应提供本地命令。
